## Project Overview

大学のプログラミング授業における課題採点をDX化するWebアプリ。学生がコードを提出し、ジャッジサーバーで自動採点、教員・TAがフィードバックを行う。

**Roles**: 学生（提出・成績確認）、教員（授業・課題管理・採点修正）、TA（採点補助）

## Repository Structure

```
backend/     # Spring Boot 3.3.5 / Java 21
frontend/    # Next.js 16 / React 19 / TypeScript
docker/      # Dockerfiles
infra/       # Terraform (AWS EKS, ap-northeast-1)
docs/        # Architecture, specs, setup guides
```

## Development Environment

### Start backend (Docker required)

```bash
docker compose -f docker-compose.dev.yml up
```

Backend runs on `http://localhost:8080` with Spring Profile `dev` and hot-reload via Spring Boot DevTools. Source `./backend` is mounted into the container.

### Start frontend

```bash
cd frontend
bun install
bun run dev     # http://localhost:3000
```

### Production build

```bash
docker compose -f docker-compose.prod.yml up -d
```

## Backend Commands

All backend commands run inside Docker in dev. For local Gradle:

```bash
cd backend
./gradlew bootRun      # Run locally (requires Java 21)
./gradlew test         # Run all tests
./gradlew test --tests "com.shootingstar.SomeTest"  # Run single test
./gradlew build        # Build JAR
```

API docs available at `http://localhost:8080/swagger-ui.html` and `http://localhost:8080/api-docs`.

## Frontend Commands

```bash
cd frontend
bun run dev       # Dev server
bun run build     # Production build
bun run lint      # Biome lint check
bun run format    # Biome format (writes)
bun run generate  # Generate API client from OpenAPI spec (backend must be running)
```

Linter/formatter: **Biome** (not ESLint/Prettier). 2-space indent, spaces only.

## Backend Architecture

Layered architecture under `com.shootingstar`:

| Package | Role |
|---------|------|
| `controller/` | `@RestController` — HTTP endpoints, `@Valid` input, Swagger annotations |
| `service/` | Business logic, `@Transactional`, authorization checks |
| `repository/` | `JpaRepository<Entity, UUID>`, custom JPQL via `@Query` |
| `entity/` | `@Entity` domain models, UUID PKs, `@Enumerated(EnumType.STRING)` |
| `dto/` | `XxxRequest` / `XxxResponse` — never expose entities directly |
| `security/` | JWT + Spring Security |

**All controllers must have `@Tag` and `@Operation` annotations** — these feed the OpenAPI spec used by the frontend code generator.

## Frontend Architecture

- **API layer**: `src/api/generated/` and `src/api/model/` are **auto-generated by orval** — do not edit manually. Regenerate with `bun run generate` after backend API changes.
- **Custom fetcher**: `src/api/fetcher.ts` — wraps `fetch`, reads `NEXT_PUBLIC_API_URL` (default: `http://localhost:8080`).
- **SWR hooks**: Generated per OpenAPI tag into `src/api/generated/`. Import and use directly in components.
- **Validation**: Zod schemas generated into `src/api/zod.generated.ts`.
- **Next.js version**: 16.2.6 — this is a newer version with breaking changes. Read `node_modules/next/dist/docs/` before writing Next.js-specific code; do not assume conventions from older versions.

## API Code Generation

When the backend API changes, regenerate the frontend client:

```bash
# 1. Start backend
docker compose -f docker-compose.dev.yml up

# 2. Regenerate
cd frontend
bun run generate
```

Generated files: `src/api/generated/`, `src/api/model/`, `src/api/zod.generated.ts`

## Git & Branch Rules

- **Branch from `main`**, PR back to `main`
- Branch naming: `<prefix>/<issue-number>-<description>` (e.g. `feat/18-add-login-api`)
- Commit format: `<prefix>: <message>` (e.g. `feat: add login api`, `fix: auth middleware bug`)
- WIP commits: `feat: (wip) implement login api`
- PR body must include `Closes #<issue-number>` to auto-close issues on merge
- All work starts from a GitHub Issue; set yourself as Assignee and move to `In Progress`

Valid prefixes: `feat`, `fix`, `refactor`, `docs`, `chore`, `test`
